# Characterization tests

Pin the CURRENT behavior of legacy code so refactors (SD-599/SD-600 and the
E-epic decompositions) can prove they changed nothing. Two kinds live here:

1. **Pure-function suites** (e.g. `includes/usdecimal/`) — no dependencies,
   run everywhere.
2. **DB-backed suites** (SD-601: `finans/` kassekladde posting and
   `includes/ordrefunc/` order→invoice) — exercise the real posting engine
   against a real PostgreSQL tenant inside the repo's docker-compose stack.
   On a machine without the stack they **skip cleanly**, so `composer test`
   stays green on a bare checkout.

## Running the DB-backed suites

Prerequisites (one-time):

```bash
docker compose up -d
# the app must be installed (includes/connect.php generated by the install
# wizard) and at least one tenant database must exist (default: saldi_2)
php tools/php/phpw.php tools/php/composer.phar install   # vendor/ (see tools/php/README.md)
```

Run inside the web container (postgres is only reachable from the compose
network — it has no host port mapping by design):

```bash
docker compose exec -T -w /var/www/html/saldi web php vendor/bin/phpunit --testsuite characterization
```

On the host the same command via `composer test` runs the pure suites and
skips the DB-backed ones with a reason.

## How the DB-backed harness works (support/)

- `CharacterizationEnv.php` — per-run bootstrap: drops and re-creates a
  throwaway tenant (`saldi_chartest`) as a `CREATE DATABASE ... TEMPLATE`
  clone of an installed tenant, registers it in master `regnskab`, and seeds
  an `online` session row (the legacy pages authenticate by session id;
  `rettigheder` is a positional digit string — see `includes/online.php:331`).
  Config via env: `SALDI_CHAR_PGHOST/PGUSER/PGPASS/MASTER_DB/TEMPLATE_DB/TEST_DB`.
- `run_bogfor_page.php` — `finans/bogfor.php` is a page script (session,
  relative includes, `exit()` branches), so it cannot be included in the
  PHPUnit process. This child-process runner drives the page's own POST
  dispatch (`$_POST['simuler']` / `$_POST['bogfor']`) exactly like a browser,
  which runs `transaktion('begin'); bogfor(...); transaktion('commit')` — the
  production code path.
- `run_order_invoice.php` — bootstraps `includes/ordrefunc.php` the way
  `remoteBooking/api.php:29-60` does in production (master connect → `$db` →
  include → tenant reconnect → derive `$regnaar`), then runs the production
  invoice sequence (`api.php:296-337`): deliver lines → `levering()` →
  `bogfor($id,'on')`.

Child processes isolate the legacy scripts' `exit()` calls; tests assert on
database state, not on page output.

## What is pinned (SD-601)

- Balanced two-account voucher: simulated posting writes balanced
  `simulering` rows, marks the kladde `S`, leaves `transaktioner` untouched.
- VAT voucher: gross 1250 on an S-coded account splits into net 1000 +
  250 VAT on the VAT account from the `SM` group.
- Real posting: balanced `transaktioner` rows, kladde marked `V`,
  `kontoplan.saldo` moves by the posted amount on both accounts.
- Unbalanced voucher: nothing reaches `simulering`/`transaktioner`.
- Order→invoice: `levering`+`bogfor` return OK, a sequential `fakturanr` is
  assigned, status ends at 4 (the re-invoice guard only requires >2),
  order sum/VAT unchanged, pending delivery quantity consumed.
- Re-invoicing an invoiced order is refused ("invoice allready created…",
  wording pinned) and the invoice number does not change.
